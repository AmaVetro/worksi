
REGLA TRANSVERSAL DE TESTING (ESENCIAL)
No se implementa testing en frontends: no carpetas ni suites de tests en el proyecto web ni en el móvil.
Quienes desarrollan frontends verifican que todo funcione de forma visual y manual en pantalla. 
La fase de pruebas estructurada se deja para el final, cuando el proyecto completo esté listo, 
y se limita a un número acotado de pruebas solo de las piezas más críticas en conjunto 
(integración entre servicios, flujo core, E2E, comprobación puntual de API si aplica). 
Se evita toda batería de testing opcional, de baja prioridad o repartida por sprint.



HU-01 — Registro de candidato
Como candidato
* Quiero registrarme en la plataforma
* Para poder postular a ofertas laborales
Criterios de aceptación:
* Se puede ingresar: nombre, apellido paterno, apellido materno, correo, contraseña, celular, rut, 
nro. documento, región y comuna como obligatorios; segundo nombre y calle como opcionales.
* Se valida formato de email.
* En creación de contraseña aplica la política alineada a Flujo Pantallas App Móvil y Contrato Unificado API: mínimo 10 caracteres; al menos una mayúscula, al menos una minúscula, al menos un símbolo y al menos un número; misma política al definir o cambiar contraseña en 
el flujo de recuperación, según esos documentos.
* Se valida lógica numérica de RUT de Chile.
* No permite emails duplicados.
* En la pantalla "Crear cuenta" no se crea aún la cuenta oficial; los datos, el PDF del CV, skills, rubro y preferencias se mantienen en estado local en el cliente hasta "Uso de datos", sin persistir candidato en el backend (salvo consumo de catálogos públicos GET /api/v1/catalogs/*), según Flujo Pantallas App Móvil y Contrato Unificado API (versión oficial del repositorio).
* La cuenta tipo "candidato" y la persistencia en BD (usuario, CV, perfil) se crean en un solo request al aceptar el consentimiento en "Uso de datos": POST /api/v1/auth/register/candidate (multipart: parte "data" JSON con consent_given=true y demás campos, parte "file" con el PDF del CV).
* Se muestra error si falla el registro
* Si falta un campo obligatorio no permite avanzar y muestra "Completa este campo"





HU-02 — Alta empresa y reclutadores (MVP web, panel administrador)
Como administrador del sistema (rol ADMIN autenticado)
* Quiero registrar la ficha de una empresa y luego crear cuentas de reclutadores vinculadas a esa empresa
* Para que existan companies en BD sin usuario “empresa”, y personas con rol RECRUITER que publiquen ofertas
Criterios de aceptación (alineados a Flujo Pantallas App Web y Contrato Unificado API):
* No hay registro público de empresa en la landing web MVP; no existe cuenta con rol COMPANY.
* Alta empresa: POST /api/v1/admin/companies (multipart; JSON en parte "data" con datos negocio: nombre comercial, razón social, teléfono, dirección, RUT, región, comuna, sector, N° trabajadores aprox.; sin email ni contraseña de empresa); imagen de organización opcional en parte "image".
* Listado de empresas: GET /api/v1/admin/companies para desplegable al dar de alta reclutadores (items con company_id, commercial_name, etc.).
* Alta reclutador: POST /api/v1/admin/recruiters con email, password (política del contrato), datos personales y company_id existente.
* Si falta un obligatorio no avanza y muestra “Completa este campo” donde corresponda.
* Con sesión RECRUITER activa en portal empresa: PATCH /api/v1/company/profile/image permite subir/cambiar/eliminar imagen en la ficha company (distincta de imagen por oferta).
* Manejo de errores visible si falla validación o servidor.
Nota: el endpoint público POST /api/v1/auth/register/company está fuera del contrato MVP web actual si no se rehabilita por acuerdo.




HU-03 — Login
Como usuario con cuenta de login (candidato CANDIDATE en app móvil; ADMIN o RECRUITER en web)
* Quiero iniciar sesión
* Para acceder a mis funcionalidades
Criterios de aceptación:
* Login con email y contraseña mediante POST /api/v1/auth/login
* Retorna token JWT válido con claim de rol según Contrato Unificado API (CANDIDATE | ADMIN | RECRUITER)
* Manejo de credenciales incorrectas
* Persistencia de sesión en cliente
* Logout elimina sesión
* En app móvil de postulantes y en web (admin/reclutador), tras 4 intentos fallidos
 se bloquea el acceso y se deriva al flujo de pantallas de recuperación de contraseña (Contrato — regla MVP v1.7+).
* En MVP, la recuperación no es real salvo la pantalla de nueva contraseña: no se genera ni se exige validar un código OTP contra el usuario; las pantallas y botones de solicitar/ingresar/reenviar código son solo maqueta (el usuario debe poder avanzar con cualquier texto en el campo código); la única parte funcional obligatoria es confirmar nueva contraseña en backend (`password_hash`); no hay envío real de correo.

HU-07 — Calcular compatibilidad (HU importante/core, priorizar)
Como candidato
* Quiero ver un porcentaje de compatibilidad con una oferta
* Para saber qué tan adecuado soy para el puesto
Criterios de aceptación:
* Se muestra porcentaje (0–100%)
* Se calcula usando IA a partir del CV limpio frente al contenido de la oferta, considerando 
además modalidades preferidas y cargas horarias preferidas del postulante
* No usa skills declaradas en el perfil
* No usa presentación personal del perfil
* No usa rango de sueldo esperado
* No usa sueldo ofrecido de la oferta
* No usa años de experiencia requeridos por la oferta
* No usa años de experiencia declarados por el postulante dentro del CV
* No usa nacionalidad del postulante
* Manejo de error si falla IA

HU-15 — Crear oferta laboral
Como reclutador (rol RECRUITER en portal empresa web)
* Quiero crear una oferta laboral
* Para atraer candidatos
Criterios de aceptación (Flujo Pantallas App Web + Contrato Unificado API):
* Campos obligatorios en creación POST /api/v1/company/jobs:
company_commercial_name (nombre comercial a mostrar; puede igualar datos de companies)
title
description (único texto largo del puesto; no existe campo persistence separado requirements; texto tipo requisitos del puesto va dentro de description)
city
region_id y commune_id coherentes con catálogo
salary_offered (obligatorio en API y modelo)
years_experience_required (obligatorio en API y modelo)
modality y workload conforme enums del contrato
skills_ids entre 3 y 8 asociadas a la oferta
* Opcional: image_url por oferta (imagen propia del aviso).
* Persistencia ACTIVE según MVP; estado único ACTIVE.
* En vistas resumidas del producto pueden mostrarse hasta 4 skills en el card; hasta 8 en detalle candidato según contrato.
* Para POST /match, job_text en backend se construye solo concatenando title, description, modality y workload; no incluye job_skills, salary_offered, years_experience_required ni company_commercial_name (solo marca en listados), según contrato.
* El porcentaje final del producto aplica también bonus por modalidades y cargas horarias preferidas del candidato frente a modality y workload de la oferta, además del score semántico base.
* Validación de campos
* Se guarda correctamente
