ARQUITECTURA OBLIGATORIA
- Frontend web (React) y frontend movil (Android) consumen solo API Spring Boot.
- Ningun frontend se conecta directo a MySQL.
- Spring Boot consume servicio IA FastAPI por HTTP interno.
- MySQL 8 es base unica del sistema.


ESTANDARES GENERALES API
- Base URL: /api/v1
- Formato: application/json salvo endpoints multipart.
- Charset: UTF-8.
- Fechas en respuestas: ISO-8601 UTC (ejemplo: 2026-04-25T03:10:00Z).
- Ids: enteros sin signo (BIGINT en BD), expuestos como number JSON.
- Moneda/sueldo: enteros (CLP) sin decimales.
- Convencion de nombres:
  - JSON de API publica: snake_case (ejemplo: first_name, region_id).
  - Codigo Java interno (entidades/DTOs/servicios): camelCase.
  - Mapeo JSON snake_case <-> Java camelCase: obligatorio mediante configuracion de serializacion o anotaciones explicitas en DTOs.
- Paginacion estandar:
  - Query params: page (base 1), size (default 20, max 100), sort (campo,asc|desc).
  - Respuesta: items, page, size, total_items, total_pages.


AUTENTICACION Y SESION
- JWT solo access token (sin refresh).
- Header en rutas protegidas:
  Authorization: Bearer <token>
- Dos familias de credencial:
  - Publicas: sin Authorization.
  - CANDIDATE / COMPANY: Bearer <access_token> emitido por POST /api/v1/auth/register/candidate, POST /api/v1/auth/login o POST /api/v1/auth/register/company segun corresponda. Claim de rol CANDIDATE o COMPANY.
- Roles en JWT de aplicacion:
  - CANDIDATE
  - COMPANY
- Logout: elimina sesion en cliente (backend no mantiene blacklist para MVP).




FORMATO DE ERRORES ESTANDAR (OBLIGATORIO EN TODA LA API)
- Formato unico:
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Hay campos invalidos",
    "details": [
      { "field": "email", "message": "Formato invalido" }

  - BUSINESS_RULE_VIOLATION
  - PAYLOAD_TOO_LARGE
  - UNSUPPORTED_MEDIA_TYPE
  - RATE_LIMITED
  - IA_UNAVAILABLE
  - INTERNAL_ERROR


ENUMS OFICIALES
- role: CANDIDATE | COMPANY
- modality: REMOTE | HYBRID | ONSITE
- workload: FULL_TIME | PART_TIME | OTHER
- job_status: ACTIVE
- application_status: APPLIED | VIEWED | CANCELLED

CONSTRUCCION OBLIGATORIA DE job_text (BACKEND)

FAIRNESS OBLIGATORIO (ANTES DE EMBEDDINGS)
- Orden: regex inicial -> spaCy NER -> regex final.
- spaCy: PERSON, ORG, GPE.
- Regex: edad, genero, nacionalidad, institucion procedencia, email, RUT y menciones de años de experiencia declarados dentro del CV.
- Solapamiento selectivo obligatorio en: nombre, institucion, nacionalidad.
- No eliminar ORG de forma indiscriminada.




ENDPOINTS DE AUTENTICACION

1) Registro candidato
- POST /api/v1/auth/register/candidate
- Auth: publica
- Momento funcional: unico request de alta, despues de que el postulante avanza por "Crear cuenta" -> (CV, skills, preferencias) en UI -> pantalla "Uso de datos" acepta consentimiento. Ahi el cliente envia de una vez identidad, password, preferencias, skills, rubro, CV y consent_given=true. No existen filas "borrador" en backend: todo se persiste en la misma operacion o falla.
- Content-Type: multipart/form-data; charset=UTF-8
- Partes (multipart, nombres oficiales):
  - "data" (string JSON, application/json en la parte): cuerpo logico con todos los campos de registro y perfil inicial (ver abajo). obligatorio.
  - "file" (binario, filename *.pdf, application/pdf): PDF del CV. obligatorio al menos en el flujo movil con pantallas que exigen subir CV antes de "Uso de datos" (mismo limite: max 1MB, solo PDF). Si se acordase flujo alterno sin CV, debe documentarse en OpenAPI; por defecto el contrato de producto exige el archivo.
- Cuerpo logico (JSON en la parte "data"):
{
  "email": "correo@dominio.cl",
  "password": "Aa!123456789",
  "first_name": "Juan",
  "middle_name": "Eduardo",
  "last_name_paternal": "Ramirez",
  "last_name_maternal": "Perez",
  "phone": "+56912345678",
  "rut": "12345678-5",
  "document_number": "12345678",
  "street": "Av. Siempre Viva 123",
  "region_id": 7,
  "commune_id": 77,
  "consent_given": true,
  "sector_id": 1,
  "profile_summary": null,
  "salary_expected_min": 700000,
  "salary_expected_max": 1200000,
  "preferred_modalities": ["REMOTE", "HYBRID"],
  "preferred_workloads": ["FULL_TIME", "OTHER"],
  "skills_ids": [10, 11, 12]
}
- Validaciones (data + file):
  - email formato valido y unico.
  - password politica obligatoria.
  - RUT valido (digito verificador).
  - comuna debe pertenecer a region.
  - consent_given debe ser true.
  - la pantalla "Uso de datos" es la ultima pantalla antes de persistir en backend: consent_given y alta ocurren juntos.
  - skills_ids entre 3 y 12; sector_id requerido segun reglas de producto; preferred_modalities y preferred_workloads coherentes con enums; rango sueldo coherente si ambos informados.
  - file: solo PDF, max 1MB, tipo application/pdf; debe coincidir con reglas de storage del backend.
- Response 201:
{
  "user_id": 101,
  "role": "CANDIDATE",
  "access_token": "jwt",
  "token_type": "Bearer",
  "expires_in": 3600
}
- Nota: el cliente mantiene en estado local (memoria de app) los datos y el archivo de CV mientras recorre las pantallas anteriores a "Uso de datos"; el backend no persiste nada de candidato hasta este POST. Si se pierde el estado (cierre forzado de app), el usuario reingresa la informacion.

MATCHING API EXTERNA (CONSUMIDA SOLO POR SPRING)

28) Match interno IA
- URL interna sugerida: /match (FastAPI)
- Request:
{
  "cv_text": "texto cv limpio",
  "job_text": "texto oferta permitido"
}
- Response:
{
  "score": 0.8215,
  "explanation": "Coincidencia en experiencia backend y herramientas"
}
- Reglas:
  - score de FastAPI es base semantica.
  - Spring transforma a 0..100 y aplica ajuste modalidad/workload.



CONTRATO DE SCORE FINAL (BACKEND -> FRONT)
- Formula fija para evitar ambiguedad:
  final_score = clamp( base_semantic_score_0_100 * 0.8 + preference_bonus_0_20, 0, 100 )
- preference_bonus_0_20:
  - +10 si modalidad oferta esta en preferred_modalities.
  - +10 si workload oferta esta en preferred_workloads.
  - +0 en cada caso si no coincide.
- Redondeo de respuesta:
  - dos decimales (DECIMAL 5,2).




CODIGOS HTTP POR TIPO DE RESULTADO
- 200: consulta/actualizacion exitosa.
- 201: recurso creado.
- 400: request invalido.
- 401: sin token o token invalido.
- 403: sin permisos sobre recurso.
- 404: recurso no encontrado.
- 409: conflicto (ejemplo: email duplicado, postulacion duplicada).
- 413: archivo excede tamano permitido.
- 415: tipo de archivo no soportado.
- 422: regla de negocio incumplida.
- 429: limite de intentos/rate limit.
- 503: IA no disponible.
